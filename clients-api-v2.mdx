## Overview

Use the Clients management API to create and manage Mythic Analytics client tenants, rotate keys, and retrieve loader snippets for embedding widgets in your applications.

This page documents the current implementation mounted under `/api/v1` in the worker, even though it appears under the `v2` Clients section in navigation.

<Callout kind="info" collapsed="false">
  The worker currently mounts all client management routes under `/api/v1`. The navigation label `Clients API v2` refers to documentation versioning, not a different runtime path.
</Callout>

## Base URL and versioning

All endpoints on this page share the same base URL and version prefix.

```text
https://api.mythic-analytics.com/api/v1
```

Combine the base URL with each path below, for example `POST https://api.mythic-analytics.com/api/v1/clients`.

## Authentication

All Clients management endpoints require an admin API key. The worker accepts the key either as a bearer token header or as a query parameter.

Use one of the following:

<ParamField header="Authorization" param-type="string" required="true" deprecated="false">
  Send `Bearer` followed by your admin key. Example: `Authorization: Bearer adm_live_9f3K2rLmQ8sPw6Dy`.
</ParamField>

<ParamField query="admin_key" param-type="string" required="false" deprecated="false">
  Alternative to the header for server-to-server or tooling scenarios. Do not use in browser-based code.
</ParamField>

<Callout kind="alert" collapsed="false">
  Never expose the admin key in client-side JavaScript, public repositories, or logs. Restrict access to trusted backend services and operations staff.
</Callout>

## Response envelope

Every endpoint returns a JSON envelope with a consistent shape.

<ParamField body="success" param-type="boolean" required="true" deprecated="false">
  Indicates whether the request succeeded.
</ParamField>

<ParamField body="data" param-type="object|null" required="false" deprecated="false">
  Resource payload for successful requests. Type depends on the endpoint (single client, list of clients, loader info, etc.).
</ParamField>

<ParamField body="count" param-type="integer" required="false" deprecated="false">
  Total number of matching resources for list endpoints.
</ParamField>

<ParamField body="error" param-type="string|null" required="false" deprecated="false">
  Error message for failed requests.
</ParamField>

Example error envelope:

```json
{
  "success": false,
  "data": null,
  "error": "Unauthorized",
  "count": 0
}
```

<Callout kind="info" collapsed="false">
  Status codes communicate high-level outcome (for example, `200`, `401`, `404`). The `success` flag and `error` string provide extra detail for programmatic handling.
</Callout>

Create a new client tenant with credentials and metadata.

### Request

**Method**

```text

```

</ParamField>

## Error handling

The Clients management API uses standard HTTP status codes alongside the envelope described earlier.

Common cases include:

- `401 Unauthorized` when the admin key is missing or invalid.

- `404 Not Found` when the specified client `id` does not exist or has been hard-deleted.

- `409 Conflict` when an operation violates a uniqueness or state constraint, such as creating a client with a duplicate `global_name` or `domain`.

<Callout kind="info" collapsed="false">
  Check both the HTTP status code and the `error` string in the response body for reliable error handling and logging across your backend and operational tooling.
</Callout>
