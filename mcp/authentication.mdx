## Management API authentication

Every endpoint under `/admin/mcp` (servers, tools, agency access, API keys) requires the platform `ADMIN_KEY`. Provide it either as a bearer token or as a query parameter.

<ParamField header="Authorization" param-type="string" required="false" deprecated="false">
  Bearer token. Format: `Bearer YOUR_ADMIN_KEY`.
</ParamField>

<ParamField query="admin_key" param-type="string" required="false" deprecated="false">
  Alternative to the header for server-to-server or tooling scenarios. Do not expose in client-side code.
</ParamField>

One of the two must be present and match `ADMIN_KEY`; otherwise the request returns `401` with `{ "error": "Unauthorized" }`.

## Gateway authentication

The gateway at `POST /mcp` accepts an `mcp_` API key created through the management API. Provide it as a bearer token or in the `X-Api-Key` header.

<ParamField header="Authorization" param-type="string" required="false" deprecated="false">
  MCP API key as bearer token. Format: `Bearer mcp_...`.
</ParamField>

<ParamField header="X-Api-Key" param-type="string" required="false" deprecated="false">
  MCP API key. Alternative to the Authorization header.
</ParamField>

The key must begin with `mcp_`. A missing, malformed, or deactivated key returns HTTP `401` with a JSON-RPC error (`code -32600`).

## Scopes

An `mcp_` key carries a `scopes` array that gates which tools it can list and call. Scopes use `resource:action` form, with `*` wildcards:

| Form | Example | Meaning |
|------|---------|---------|
| `resource:action` | `insights:read` | One action on one resource |
| `resource:*` | `dashboards:*` | Any action on one resource |
| `*` | `*` | Any tool |

<Callout kind="info">
  An **empty** `scopes` array grants unrestricted access to every tool the key's agency and location can reach. Grant explicit scopes to narrow a key down.
</Callout>

<Callout kind="info">
  The same `mcp_` keys authenticate the [Settings Server](/mcp/settings-server),
  which adds the `destinations`, `tags`, `snippets`, and `transformers` resources
  and supports **agency-wide keys** (no fixed location â€” they select a client per
  session). A key with a `null` location is agency-wide; one bound to a location
  is client-scoped.
</Callout>

### How tool names map to scopes

Built-in tools are mapped by name prefix, and the resource is normalized to `insights`, `dashboards`, or `canvases`:

- `list_*`, `get_*` require `<resource>:read`
- `create_*`, `update_*`, `delete_*`, `add_*`, `remove_*` require `<resource>:write`

Upstream tools require `<scope_prefix>:call`, where `<scope_prefix>` is the owning server's configured prefix. If the server has no scope prefix, the required scope is `upstream:call`.

<Callout kind="alert">
  Never expose the admin key or `mcp_` API keys in client-side JavaScript, public repositories, or logs. Both are server-side credentials. The raw `mcp_` key is shown only once, at creation.
</Callout>
