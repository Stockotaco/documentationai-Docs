## Overview

The MCP Gateway is a single JSON-RPC 2.0 endpoint. It authenticates the caller's `mcp_` key, enforces scopes and agency/location access, then either runs a **built-in tool** directly against Supabase or proxies the call to a registered **upstream** MCP server (injecting that client's credentials).

## Endpoint

```text
POST https://mythic-analytics.gulp.workers.dev/mcp
```

The endpoint accepts a single JSON-RPC request object per HTTP call. CORS is open (`*`) for `POST` and `OPTIONS`.

## Authentication

Send the `mcp_` key as a bearer token or in `X-Api-Key`.

```text
Authorization: Bearer mcp_...
```

See [Authentication](/mcp/authentication) for scope rules.

## JSON-RPC 2.0 format

### Request

```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "method": "tools/call",
  "params": {
    "name": "list_insights",
    "arguments": { "limit": 10 }
  }
}
```

<ParamField body="jsonrpc" param-type="string" required="true" deprecated="false">
  Must be `"2.0"`.
</ParamField>

<ParamField body="id" param-type="string|number|null" required="false" deprecated="false">
  Request identifier, echoed back in the response.
</ParamField>

<ParamField body="method" param-type="string" required="true" deprecated="false">
  One of `initialize`, `notifications/initialized`, `tools/list`, `tools/call`.
</ParamField>

<ParamField body="params" param-type="object" required="false" deprecated="false">
  Method-specific parameters. For `tools/call`, include `name` and `arguments`.
</ParamField>

### Success response

```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "result": {
    "content": [
      { "type": "text", "text": "[ ... insights ... ]" }
    ]
  }
}
```

### Error response

```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "error": {
    "code": -32601,
    "message": "Method not found: tools/execute"
  }
}
```

## Methods

### initialize

Returns protocol and server info. No authentication side effects beyond the standard key check.

```json
{ "jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {} }
```

Result:

```json
{
  "protocolVersion": "2025-03-26",
  "capabilities": { "tools": {} },
  "serverInfo": { "name": "mythic-analytics-gateway", "version": "1.0.0" }
}
```

### notifications/initialized

Client lifecycle notification. The gateway accepts it and returns an empty result.

### tools/list

Returns every tool the key can use — built-in tools allowed by scope, plus upstream tools allowed by scope, agency access, and location access. Each entry has `name`, `description`, `inputSchema`, and `annotations`.

```json
{ "jsonrpc": "2.0", "id": 2, "method": "tools/list", "params": {} }
```

### tools/call

Execute a tool by name.

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "create_dashboard",
    "arguments": { "name": "Q3 Overview" }
  }
}
```

The result is the tool's MCP content payload (`{ "content": [...] }`, optionally with `"isError": true`). Built-in tools return their content as a JSON string inside a text block. Upstream tool results are proxied through unchanged.

<Callout kind="info">
  `location_id` is injected from the API key for built-in tools — you do not pass it when creating resources. Upstream calls receive that client's credentials automatically.
</Callout>

## Built-in tools

These tools are always implemented by the gateway (subject to scope). They operate on the key's location.

| Tool | Scope required |
|------|----------------|
| `list_insights` | `insights:read` |
| `get_insight` | `insights:read` |
| `create_insight` | `insights:write` |
| `update_insight` | `insights:write` |
| `delete_insight` | `insights:write` |
| `list_dashboards` | `dashboards:read` |
| `get_dashboard` | `dashboards:read` |
| `create_dashboard` | `dashboards:write` |
| `update_dashboard` | `dashboards:write` |
| `delete_dashboard` | `dashboards:write` |
| `add_insight_to_dashboard` | `dashboards:write` |
| `remove_insight_from_dashboard` | `dashboards:write` |
| `list_canvases` | `canvases:read` |
| `get_canvas` | `canvases:read` |

Any additional tools come from registered upstream servers and require `<scope_prefix>:call` (or `upstream:call`).

## Error codes

Standard JSON-RPC codes are used throughout:

| Code | Meaning |
|------|---------|
| `-32700` | Parse error — invalid JSON |
| `-32600` | Invalid request — bad envelope, auth failure, or access/scope denial |
| `-32601` | Method not found, or unknown tool |
| `-32602` | Invalid params — missing tool name |
| `-32603` | Internal error |
| `-32000` | Rate limit exceeded, or an upstream transport/HTTP error |

<Callout kind="info">
  Authentication failures return HTTP `401` and rate-limit rejections return HTTP `429`, each carrying a JSON-RPC `error` object. Scope and access denials are returned as `-32600`.
</Callout>

## Rate limiting

The gateway enforces a per-key, per-minute request ceiling using a sliding window. The limit comes from the key's `rate_limit_per_minute` (default `60`). Exceeding it returns HTTP `429`:

```json
{ "jsonrpc": "2.0", "id": null, "error": { "code": -32000, "message": "Rate limit exceeded" } }
```

<Callout kind="alert">
  Limits are per API key, not per location. Provision separate keys for separate services to isolate their limits.
</Callout>
